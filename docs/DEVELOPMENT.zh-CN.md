# 中国思想图谱开发文档

版本：0.1  
日期：2026-07-13  
状态：第一阶段观点聚焦交互完成，观点数据扩充进行中（Schema v4）

## 1. 项目目标

构建一套从传统上古叙事延伸到当代中国哲学研究者的“思想命题关系时轴”。系统以命题为最小知识节点，以时间组织人物和命题，以关系连接不同历史时期的观点，并允许通过表格、后台或 API 持续更新数据后自动重绘。

UI 功能逻辑参照 History of Philosophy：全屏画布、斜向时间轴、高密度命题、正负关系曲线、人物/命题聚焦、组合筛选和动态 URL。实现必须使用本项目自己的代码、品牌、素材与数据。

## 2. 三阶段路线

### 第一阶段：人物科研基线与时间轴显示系统

目标是同时建立可靠的图形交互内核与可公开审查的人物基础名录。人物纳入必须满足 `RESEARCH-METHOD.zh-CN.md`，不得以知名度或孤立名言替代研究证据。

必须实现：

1. 左上向右下的连续内容流；出生年只决定排序和年代显示，不映射为比例距离。
2. 不绘制实体时间轴或年代刻度；朝代与历史时期通过人物年代、筛选器和详情呈现。
3. 人物作为观点体系与时间容器；人物下属的命题才是知识节点。
4. 正向语义关系和负向语义关系曲线。
5. 独立的方向字段，用于表达明确影响、继承或传播，不能把颜色误当方向。
6. 缩放、平移、适合屏幕和视口恢复。
7. 人物聚焦、命题聚焦、一度关系和进一步关系展开。
8. 人物、思想主题、时期、关系类型和入门命题筛选。
9. URL 保存视图、筛选和聚焦状态。
10. 中文、英文完全等价的页面和 API 响应。
11. CSV 驱动的中英人物、命题、关系、分类和来源数据，更新后自动计算位置并重建页面。
12. 分类共现只生成后台研究候选，绝不作为公开连线；公开连线必须连接两条已审核命题。
13. 可考、传统、文本人格、神话—传说人格的明确区分。

渲染演进：

- 原型阶段使用 SVG，便于验证内容槽位、曲线和交互。
- 节点超过约 1,000 或边超过约 5,000 后切换 Canvas/WebGL。
- 大规模布局、筛选和空间索引放入 Web Worker。
- 采用连续层级细节：人物标题和命题文字始终存在，总览时随画布缩为短线纹理，放大后恢复为可读内容。

第一阶段验收标准：

- 100、1,000 和 10,000 节点基准数据均有性能报告。
- 人物顺序由出生年字段计算，位置由人物槽与观点槽自动累积，不允许手工坐标成为权威数据。
- 所有筛选可组合并能通过 URL 恢复。
- 键盘、屏幕阅读器和减少动态效果模式可用。
- 中英文界面通过 `npm run i18n:check`。

### 第二阶段：命题与关系数据扩充和精校

人物基础名录和观点关系内核已在一期建立。二期继续从原典或可靠研究中扩充人物命题，并逐条精校命题之间的相似、扩展、反驳、对照、影响和传播关系。关系默认是无向的概念关系；只有存在证据时，才能另行标记历史影响或传播。

“全部人物”被定义为持续扩展的开放名录，不声称一次性穷尽。人物进入公开图谱需要经历：候选、初审、资料齐备、关系审核、公开五个状态。

数据搜集顺序：

1. 建立朝代、时期、学派、机构和地域控制词表。
2. 从哲学史通史、断代史、学术史和权威辞典生成第一批人物候选。
3. 补充经学家、宗教思想家、政治思想家、科学思想家、文学思想家及海外华人学者。
4. 为每个人建立权威名称、别名、生卒年、活动年代、身份、作品和来源。
5. 从原典或可靠研究中拆分命题，禁止用人物简介自动伪造命题。
6. 对命题关系逐条记录证据、判断者、置信度和历史影响性质。
7. 中英文内容分别记录完成状态、译者、审核者和更新时间。

来源优先级：

1. 权威原典校注本、全集和学术机构数字资源。
2. 正规出版社哲学史、思想史和人物研究专著。
3. 同行评议论文、大学或研究机构人物页。
4. 百科与开放数据仅用于发现线索，不能作为关键关系的唯一证据。

### 第三阶段：维护与管理系统

后台必须支持：

- XLSX/CSV 导入、字段校验和差异预览。
- 人物、命题、主题、关系和来源的增删改查。
- 中英文并排编辑与翻译状态追踪。
- 草稿、待审、通过、退回和撤下工作流。
- 关系证据审核和冲突提示。
- 批量标签、批量时期映射和别名合并。
- 发布版本、操作日志、数据快照和回滚。
- 数据更新后按标签失效缓存并自动刷新前台图谱。
- 导出可机器读取的数据包和研究引用版本。

## 3. 系统架构

```text
浏览器交互层
  ├─ 时间轴 Canvas/WebGL
  ├─ HTML 筛选与详情面板
  └─ URL 状态
        ↓
Next.js App Router
  ├─ Server Components：初始数据与 SEO
  ├─ Client Components：画布与交互
  ├─ Route Handlers：公开 API、导入和导出
  └─ Server Functions：后台写操作
        ↓
领域与数据访问层
  ├─ 人物（容器）、命题（节点）、关系（命题边）、主题、来源
  ├─ 国际化解析与缺失语言报告
  └─ 图聚合、筛选和视口查询
        ↓
PostgreSQL + 对象存储 + 可选 Redis
```

数据库客户端和外部服务必须延迟初始化，不能在模块顶层创建。公开读接口与后台写接口分开；代理层不能成为唯一权限边界。

## 4. Ubuntu 部署

生产环境采用 Docker 多阶段构建和 Next.js standalone 输出。推荐拓扑：

```text
Internet → nginx/Caddy → Next.js container → PostgreSQL
                                  └──────→ Redis（规模化后）
```

要求：

- Ubuntu LTS、Docker Engine、Compose v2。
- nginx/Caddy 负责 TLS、限流、请求体限制和异常连接防护。
- 容器使用非 root 用户。
- 健康检查访问 `/api/health`。
- 单实例阶段使用持久磁盘缓存；多实例阶段使用共享缓存和统一部署 ID。
- 数据库每日备份，导入前自动创建数据快照。
